# Local-first `.md` Files — 設計仕様書

**作成日**: 2026-05-13
**ステータス**: ブレインストーミング承認済み・実装計画待ち
**前提**: 現行は Next.js (Web) + Tiptap + Zustand + localStorage の Notion 風ブロックエディタ

---

## 1. 目的

ドキュメントの永続化先を localStorage から **ユーザーのファイルシステム上の `.md` ファイル** へ移行する。これにより：

1. **データ主権**：ドキュメントはユーザーの手元に存在し、サーバーを経由しない（既に達成しているが、より明確化）
2. **消失リスク低減**：localStorage クリアでデータが消えるリスクを排除

非目標：クラウド同期、複数ワークスペース、サブフォルダ、Git 連携、検索、タグ、リンク、frontmatter、外部編集追従、PDF エクスポート。

---

## 2. アーキテクチャ全体像

### スタック

| レイヤ | 採用 |
|---|---|
| デスクトップシェル | **Tauri 2** |
| フロントエンド | 既存の Next.js を `output: 'export'` で **静的書き出し**、Tauri がロード |
| エディタ | 既存の Tiptap（変更なし） |
| 状態管理 | Zustand（用途を縮小、後述） |
| 永続化 | **ファイルシステム**（localStorage 廃止） |

### モジュール責務

| レイヤ | ファイル | 責務 |
|---|---|---|
| Tauri (Rust) | `src-tauri/src/` | ディレクトリ選択ダイアログ、`.md` 読み書き、`.stash/` 管理、パス検査 |
| TS FS ラッパー | `src/lib/fs/` | `listMarkdownFiles`, `readDocument`, `writeDocument`, `renameDocument`, `deleteDocument`, `listStash`, `stashDocument`, `restoreStash`, `deleteStash` |
| Markdown 変換 | `src/lib/markdown/` | 既存 `tiptapToMarkdown` + 新規 `markdownToTiptap`（MD→Tiptap JSON） |
| ワークスペース状態 | `src/store/workspaceStore.ts` | ワークスペースパス、ファイル一覧、アクティブドキュメント |
| エディタ状態 | `src/store/editorStore.ts` | 編集中ドラフトと未保存フラグ（既存を縮小） |

### 起動フロー

1. アプリ起動 → 直近のワークスペースパスを Tauri Store から読み込み
2. パスが存在して有効 → そのまま Zen ページへ
3. パスが無効 / 未設定 → ウェルカム画面でフォルダ選択ダイアログ
4. フォルダ選択完了 → 直下の `.md` を列挙し、**最も最近編集されたファイル**を Zen で開く（無ければ空のドラフト）

## 3. ワークスペースとファイル管理

### ディレクトリ構成

```
~/Documents/Zikon/           ← ユーザー選択のワークスペース
├── 設計メモ_abc123.md
├── 議事録_xyz789.md
├── untitled_def456.md
└── .stash/                  ← 隠しフォルダ（Stash 配下）
    └── ボツ案_old001.md
```

`.stash/` 以外の Zikon 管理用名前空間は導入しない。将来必要になった時点で別名を決める。

### ファイル名規則

- フォーマット：`{H1タイトル}_{ID}.md`
- **ID**：6 文字の base36（`Date.now()` を base36 化した末尾 6 文字）
- H1 未入力時：`untitled_{ID}.md`
- H1 のサニタイズ：`/ \ : * ? " < > | 改行 タブ` を `_` に置換
- 長さ制限：H1 部分は先頭 40 文字まで（拡張子と ID を含めて Windows の 255 文字制限内）

### ファイル形式

- **純粋な Markdown のみ**。frontmatter（YAML ヘッダ）は採用しない
- 既存 `tiptapToMarkdown` で書き出し
- 開く時は新規実装の `markdownToTiptap`（`unified` + `remark-parse` ベース）で読み込み
- 文字コードは **UTF-8 固定**、改行は **`\n` 固定**

### ファイル衝突

- ID 付きなので物理的衝突は発生しない
- 同名 H1 + 別 ID が並ぶのは正常（Kura で両方表示）

### ワークスペース切替

- サイドバー下部に「ワークスペースを変更」リンク
- 切替時：現在の未保存変更を flush → 新ワークスペースをロード

## 4. エディター動作

### タイトルの扱い

- **タイトル入力欄を削除**（既存 UI の変更）
- エディタ冒頭の **最初の H1** がタイトル
- 新規ドキュメントは空の H1 ブロックで始まる（プレースホルダ「無題のタイトル」）
- H1 が複数ある場合、**最初に登場する H1** のみファイル名に使用
- H1 が無い / 空の場合、ファイル名は `untitled_{ID}.md`

### 自動保存とリネーム

- 既存の auto-save（600ms デバウンス）を踏襲、書き出し先を **ファイル** に変更
- 同じデバウンスで H1 変更検知時にファイル名も追従リネーム
- 書き込み順序：
  1. （リネームがある場合）旧パス → 新パスへ `rename`
  2. 新パスに最新の Markdown を書き込み

### 切替時の保存保証

- 別ファイルを開く / アプリ終了時、保留中のデバウンスを即座に flush（強制保存）
- 失敗時はエラートースト + 保留状態を維持

### 既存機能（変更なし）

- ブロック構造、スラッシュコマンド、バブルメニュー、コードブロック Tab 挙動、ダーク濃紺テーマはそのまま

### 新規アクション

- **`/stash`**：現在の編集対象を `.stash/` へ mv → エディタを新規空ドキュメントに
- **新規ドキュメント作成**：コマンドメニュー / サイドバー / Kura 上部のボタンから実行可。ワークスペース直下に空の `.md` を作成

---

## 5. Stash 機能

### ストレージ

- 物理場所：`{workspace}/.stash/`
- ファイル名規則：通常ドキュメントと同じ `{H1}_{ID}.md`

### 操作

| アクション | 動作 |
|---|---|
| `/stash`（スラッシュコマンド） | 現在のドキュメントを `.stash/` へ `fs::rename` → エディタを空ドキュメントに |
| ヘッダの Stash ボタン | プルダウンに `.stash/` 配下の `.md` をリスト |
| 復元 | `.stash/` → ワークスペース直下へ `fs::rename` → そのファイルを Zen で開く |
| 削除 | `.stash/` のファイルを `unlink` |

### 移動セマンティクス

- 「コピー＋削除」ではなく **`fs::rename`（mv）** に統一
- 退避と復元はフォルダの違いだけ

### 空ドキュメントの退避抑制

- タイトル空 + 本文空のとき、退避はせず単に新規空ドキュメントに置き換え

### Stash 内ファイルの編集

- v1 では Stash 内ファイルは **閲覧（プレビュー）のみ**
- 編集は必ず復元してから実施

### ディレクトリ作成

- `.stash/` が無ければ初回 Stash 操作時に on-demand 作成

## 6. Kura（ファイル一覧）

### 構成

- 既存の `/kura`（プレースホルダ）を本実装に置き換え
- ワークスペース直下の `.md` を一覧表示（`.stash/` を含む隠しフォルダは除外）

### 表示項目（各行）

- **タイトル**：ファイル先頭の H1（先頭 4KB を読んで正規表現 `^# .+` で抽出）
- **ファイル名**：副表示（例 `設計メモ_abc123.md`）
- **更新時刻**：OS の mtime を相対表示（「3 分前」「昨日」「2024-12-15」）

### 並び順

- デフォルト：**更新時刻の新しい順**
- ソート切替は v1 スコープ外（後付け可能な抽象を残す）

### 操作

| 操作 | 動作 |
|---|---|
| 行クリック | そのファイルを Zen で開く（現在開いているファイルは保存 flush） |
| 新規作成ボタン | 空の `.md` を作成して Zen へ。`untitled_{ID}.md` |
| ホバー時の削除ボタン | 確認モーダル → unlink + 一覧から消す |
| 再読み込みボタン | ワークスペースを再スキャン |

### 削除

- v1 は **完全削除**（ゴミ箱送りなし）
- 将来 `.trash/` への mv へ拡張可能な抽象を保つ

### 空ワークスペース時

- 「まだドキュメントがありません」+「新規作成」ボタンを大きく表示

---

## 7. 外部編集（非対応）

- **ファイル監視を実装しない**（Rust の `notify` 依存も入れない）
- 競合ダイアログ、差分表示は無し
- Kura のファイル一覧は **ページマウント時に `readdir`**（毎回最新）
- 編集中ファイルを外部から書き換えても検知しない（次に Kura に戻った時に間接的に反映される）
- 編集中ファイルを外部から削除されても Zikon は気にせず、次の保存タイミングで再作成する
- Kura ヘッダに **手動の再読み込みボタン**だけ用意

---

## 8. マイグレーション（不要）

- localStorage の既存データは **無視**（クリアもしない、放置）
- 初回起動から `.md` ファイル運用一本

## 9. テスト・非機能要件

### テスト方針

**TypeScript 側ユニットテスト**：

- `tiptapToMarkdown` / `markdownToTiptap` のラウンドトリップ（変換 → 逆変換で意味的に同等）
- ファイル名サニタイズ（特殊文字 / 長さ制限）
- ID 生成（6 文字 base36、衝突しないこと）

**Rust 側テスト**：

- `read_document` / `write_document` / `rename_document` / `delete_document` の正常系
- 不正パス拒否（`../../etc/passwd` のようなワークスペース脱出）

**E2E**：v1 では入れない。手動シナリオ確認のみ。

### セキュリティ

- Tauri の `fs` allowlist を **ワークスペース配下にスコープ制限**
- パス検査：ワークスペースをエスケープするパスは Rust 側で reject
- 書き込みは UTF-8 固定、改行 `\n` 固定

### パフォーマンス

- Kura のタイトル抽出は **先頭 4KB** のみ読み込む
- ファイル一覧は同期的列挙（ファイル数 100 程度を想定。それ以上は v2 検討）

### 配布

- v1 は macOS のみ（Apple Silicon + Intel ユニバーサル）
- 署名・公証は v1 スコープ外（自己ビルド前提）
- Tauri なので Linux/Windows 対応は後付け容易

## 10. 非対応（v1 で明示しないもの）

- 複数ワークスペース
- サブフォルダ / 階層

- 外部編集追従 / 競合解決
- PDF / HTML エクスポート
